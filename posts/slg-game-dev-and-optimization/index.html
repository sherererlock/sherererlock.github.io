SLG 手游的大地图制作与性能优化实战

March 25, 2026 · shlock

前言#

本文整理自一次技术分享，内容基于一款 3D 写实大型魔幻 SLG 手游的开发经验。该游戏包含大地图、日夜系统、实时点光源、家园系统等特性，运行在移动设备上。

一、SLG 手游的特点#

SLG 游戏有几个对开发很有影响的特点：

  1. 地图尺寸大、地形丰富：既要表达庞大的地图，又要控制包体大小，尤其是海外发行时对包体要求很高
  2. 层次分明、顺序统一：渲染时可以按固定顺序——先画建筑、再画树、再画地形、最后画水，不需要复杂的排序
  3. 物体多、重复度高：非常适合实例化（GPU Instancing）技术
  4. 建筑大多只有位移，很少旋转：存储时只需存 position，不需要存完整的 matrix，极大节省存储

二、大地图制作#

地形数据流程#

使用 Unity 自带的 Terrain 编辑器，经过三级加工：

  1. 美术制作原始地形数据
  2. 放置树木、建筑等装饰物
  3. 策划配置功能建筑的 Resource Table
  4. 编译机自动整合：修整地形、调整装饰物位置、输出最终地形数据和高度图

高度图压缩#

在 32 位、16 位、8 位三种方案中选择了 8 位：

  • 经测试，在 0~25 米落差范围内，8 位高度图的精度足够光滑
  • 美术确认这个范围能满足大部分表达需求
  • 相比 16 位，存储和内存节省显著

河道和峡谷处理#

高度图有一个天然缺陷：表达蜿蜒曲面（如河道）时会出现锯齿。

解决方案：自适应地形细分

  • 将地形分为 9 个区域，计算每个区域的高度梯度
  • 梯度大的区域（起伏多）→ 给更多顶点
  • 梯度小的区域（平坦）→ 给较少顶点，节省性能

地形抹平#

静态抹平：在编辑器中离线完成，在建筑放置区域自动平整地形，并加上曲线过渡使其自然。

动态抹平：运行时实时计算

  • 用一张 64×64 的贴图，每个像素对应游戏中的一个逻辑格（5×5 米）
  • 像素值表示该位置是否有建筑
  • 实时计算出平滑的高度过渡

地表渲染优化（RVT）#

传统做法：4~16 层纹理混合，4 层就需要 9 张贴图，低端机（红米 Note 8）上要 30ms。

改用 Runtime Virtual Texture（RVT）方案：

  • 将所有地表层的混合结果预先渲染到两张虚拟纹理：Normal VT + Diffuse VT
  • 每帧只需生成 1~2 次 VT，之后所有光照计算都从这两张纹理读取
  • 性能提升：32ms → 2.5ms（红米 Note 7），10 倍以上

地表贴花（Deco）#

道路上的路、陨石坑、联盟边界线等需要画在地表上。

传统做法是用 Decal 投影，但每帧做矩阵投影计算太贵。

改进方案：把 Decal 直接画到 RVT 上

  • Decal 先画到一张中间贴图（Decal VT）
  • 再和地形 VT 混合，得到最终效果
  • 联盟边界等功能性 Decal 会变化——直接在 Decal VT 层更新，不用重新混合全部地形
  • 常态下性能开销为 0，变化时仅 1.58ms

三、水体处理#

两个关键问题：深浅效果 和 接缝处理。

方案：直接使用场景深度图实时计算

  • 只需读取屏幕 1/4 大小的深度图，带宽很小
  • 好处：水面的深浅效果实时计算，支持任意形状的水底地形
  • 拿一个 Z 字形曲面就能直接当瀑布用，深浅效果自然

四、性能优化#

Shader Map 改进#

传统 Shadow Map 的问题：利用率低。生成阴影的摄像机视角和玩家看到的画面范围不重合，大量 Shadow Map 区域对当前画面没有贡献。

改进方案：基于 Screen Space 的 Shadow Map

  • 把建筑物直接拍平，只写入顶点的 Y 值
  • 和最终屏幕像素基本一一对应，利用率远高于传统方案
  • 同等精度下，传统方案需要 4096×4096 的 Shadow Map，新方案只需一个 Screen 大小
  • 省掉了两个矩阵乘法，红米 8 上从 15.96ms 降到 14.12ms，节省约 1.8ms

多光源优化#

日夜系统加入了实时点光源，这在 SLG 中比较少见。

传统方案的痛点：

  • 需要三张光源列表（Global List、Light Index、Light Group），内存大
  • 依赖 Compute Shader + SSBO，兼容性差——很多低端机"好像支持但实际不可用"

改进方案：用位掩码替代 SSBO

  • 用一个 32 位整数，每一位代表一盏灯——一个 Tile 只需一个整数
  • Light Culling 在 Fragment Shader 中完成，结果存在 INT 类型贴图中
  • 完全绕开 SSBO 和 Compute Shader，兼容性大幅提升
  • 同屏 100 盏灯时，内存占用比 Unity URP 和 Orange 那套都低
  • 对低端机很友好——可能是低端机唯一能支撑多点光源的方案

粒子系统重构#

问题：大型战斗中同时存在 1500+ 个 Particle System，虽然单个粒子很简单（两个三角形），但合批效率低。

重构思路：

  • 同材质的不同 Particle System 可以合并绘制，不需要按实例分开
  • 重构后的粒子系统在三星 Galaxy S8 上对比原生方案有明显性能优势，且粒子越多差距越大

运行时纹理压缩#

项目后期面临内存和带宽压力。

方案：运行时对大纹理（VT、RT 等）进行 ETC/ASTC 压缩

  • 目标：在内存和带宽之间找到平衡
  • 计划最终采用 ASTC，兼容性更好（目前还在开发中）

五、移动端发热控制#

经过半年的研究，发热问题归结为两个核心因素：

1. 带宽#

带宽直接影响温度。实测数据表明：带宽控制在 2GB/s 以内，手机温度基本可控；超过 2.3GB/s 就很难控制了，最终导致 CPU/GPU 降频和卡顿。

这就是为什么一定要做运行时纹理压缩。

2. CPU 单核利用率#

在一些"看似没压力"的场景（如登录界面）反而严重发热——原因是 CPU 单核利用率达到 100%。

多线程调度时要注意：某个线程如果把单核跑满，发热就会飙升。

总结：控制好带宽和单核利用率这两点，移动端发热问题基本可控。

总结#

本文从实际项目出发，覆盖了 SLG 手游从大地图制作到性能优化的全流程：

模块 核心技术 性能收益
高度图 8 位压缩 存储减半
地形细分 自适应梯度 节省 30%+ 顶点
地表渲染 RVT 方案 32ms → 2.5ms
阴影 Screen Space Shadow Map 利用率大幅提升
多光源 位掩码 + Fragment Culling 绕开 SSBO，低端机可用
粒子系统 跨实例合批 粒子越多优势越大
发热控制 带宽 + 单核利用率 可控温
